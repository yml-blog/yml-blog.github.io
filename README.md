# yangmingli.com

Static HTML/CSS/JS personal site and blog for Yangming Li.

## SEO Maintenance

Run the audit before publishing structural or metadata changes:

```bash
python scripts/seo_audit.py
```

Regenerate the sitemap after adding, removing, renaming, canonicalizing, or noindexing pages:

```bash
python generate_sitemap.py
```

The sitemap should contain only canonical, indexable, existing HTML pages. `robots.txt` should continue to point to `https://yangmingli.com/sitemap.xml`.

## Newsletter Setup

The newsletter is intentionally lightweight: static forms submit to Vercel serverless functions, subscriber state lives in Supabase, and newsletter emails are sent through Resend from a local script or GitHub Actions workflow.

### Database

Apply the Supabase migration:

```bash
supabase db push
```

Or paste `supabase/migrations/20260615_newsletter_subscribers.sql` into the Supabase SQL editor. The table is `public.newsletter_subscribers` with:

- `id`
- `email`
- `status`: `subscribed`, `unsubscribed`, or `pending`
- `source`
- `created_at`
- `updated_at`
- `unsubscribed_at`

RLS is enabled. Do not add public read/write policies for this table; access should go through the server-side API using the Supabase service role key.

### Environment Variables

Configure these in Vercel and in any local shell that sends newsletters:

```bash
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
NEWSLETTER_SIGNING_SECRET=use-a-long-random-secret
NEWSLETTER_SITE_URL=https://yangmingli.com
RESEND_API_KEY=re_...
NEWSLETTER_FROM="Yangming Li <newsletter@yangmingli.com>"
NEWSLETTER_REPLY_TO=dleeym95@gmail.com
RESEND_SEGMENT_ID=optional-resend-segment-id
NEWSLETTER_ALLOW_ORIGIN=https://yangmingli.com
NEWSLETTER_RATE_LIMIT_MAX=8
NEWSLETTER_RATE_LIMIT_WINDOW_MS=600000
```

Never expose `SUPABASE_SERVICE_ROLE_KEY`, `RESEND_API_KEY`, or `NEWSLETTER_SIGNING_SECRET` in frontend JavaScript.

### Subscribe and Unsubscribe

Forms with `data-newsletter-form` post JSON to `/api/newsletter-subscribe`. The API validates email format, uses a honeypot field, rate-limits basic spam attempts, prevents duplicate subscriptions, and returns friendly JSON responses.

Every newsletter email must include the per-subscriber unsubscribe URL generated by `buildUnsubscribeUrl`. The unsubscribe endpoint is `/api/newsletter-unsubscribe?id=...&token=...` and marks the subscriber as `unsubscribed`. Sending scripts query only `status=eq.subscribed`, so unsubscribed readers are excluded.

### Sending a Newsletter

Dry-run the latest RSS digest:

```bash
node scripts/send_newsletter.js --dry-run --since-days 7 --preview-out newsletter-preview/digest
```

Send a digest for recent RSS posts:

```bash
node scripts/send_newsletter.js --send --since-days 7
```

Send a newsletter for a specific newly published post:

```bash
node scripts/send_newsletter.js --send --post uplift-modeling-industry-guide.html
```

Use `--subject` and `--intro` to customize the email:

```bash
node scripts/send_newsletter.js --send --post my-new-post.html --subject "New technical note: ..." --intro "A short note on what changed and why it matters."
```

The optional GitHub Actions workflow at `.github/workflows/newsletter-digest.yml` runs weekly. By default it creates a preview artifact. It sends only when manually dispatched with `send=true` or when the repository secret `NEWSLETTER_SEND_WEEKLY` is set to `true`.

### Newsletter Tests

Run the lightweight newsletter tests:

```bash
node --test scripts/test_newsletter_core.js
```

## API Key Restrictions

The homepage loads Google Maps only when the map area is needed. The Maps JavaScript API key used by the static frontend must be restricted in Google Cloud Console before deployment:

- Application restriction: HTTP referrers limited to `https://yangmingli.com/*` and any intentional preview domains.
- API restriction: Maps JavaScript API only.
- Do not commit or deploy an unrestricted browser key.
